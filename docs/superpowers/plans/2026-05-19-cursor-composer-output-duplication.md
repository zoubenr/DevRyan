# Cursor Composer Output Duplication Fix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stop duplicated final assistant output from Cursor Composer streams without changing thinking/reasoning output behavior or suppressing legitimate repeated text.

**Architecture:** The bug is in the assistant `text` part data path, not the reasoning renderer. Cursor ACP/Composer can emit repeated full text/output deltas or repeated adjacent text snapshots; DevRyan currently concatenates text deltas in the hot sync reducer and renders adjacent text parts verbatim. Fix this with narrow normalization at the sync boundary for exact duplicate delta frames and at render-time for exact duplicate adjacent assistant text blocks, while keeping reasoning parts untouched.

**Tech Stack:** Bun tests, TypeScript/React, Zustand sync reducer, OpenCode SDK event parts.

---

## Root Cause Analysis

The duplicated line in the screenshot is rendered as bright assistant output, not dim reasoning. That maps to `part.type === "text"` in `packages/ui/src/components/chat/message/MessageBody.tsx` and `AssistantTextPart.tsx`, not `ReasoningPart.tsx`.

Current pipeline:

1. OpenCode/Cursor sends `message.part.delta` events.
2. `packages/ui/src/sync/event-pipeline.ts` coalesces consecutive deltas for the same `(messageID, partID, field)` by concatenating `prev.properties.delta + inc.delta`.
3. `packages/ui/src/sync/event-reducer.ts` applies `message.part.delta` by appending the delta to the current field value.
4. `MessageBody.tsx` merges consecutive assistant text parts by joining their extracted text with `\n`.

The existing reducer guard only handles this pattern:

```text
part.updated(text="Checking the current file state and finishing the move: ")
part.delta(delta="Checking the current file state and finishing the move: ...")
```

It does not handle either of these Cursor Composer patterns:

```text
part.delta(delta="Checking the current file state and finishing the move: ...")
part.delta(delta="Checking the current file state and finishing the move: ...")
```

or:

```text
text part A = "Checking the current file state and finishing the move: ..."
text part B = "Checking the current file state and finishing the move: ..."
```

The malformed `edit` tool messages are not generated by DevRyan strings; repository search found no `Skipped malformed tool call`, `Tool loop guard`, or `missing required: old_string` UI copy in DevRyan. Those lines are provider/OpenCode output being surfaced as assistant text. The duplicated/jammed display is still fixable in DevRyan by normalizing exact duplicate output chunks before rendering.

Replace the file contents with:

```typescript
export function appendNonOverlappingDelta(existingValue: string | undefined, delta: string) {
  if (!existingValue || delta.length === 0) return (existingValue ?? "") + delta
  if (existingValue.endsWith(delta)) return existingValue

  const maxOverlap = Math.min(existingValue.length, delta.length)
  for (let overlap = maxOverlap; overlap > 0; overlap--) {
    if (existingValue.endsWith(delta.slice(0, overlap))) {
      return existingValue + delta.slice(overlap)
    }
  }

  return existingValue + delta
}

export function appendStreamingTextDelta(existingValue: string | undefined, delta: string) {
  const existing = existingValue ?? ""
  if (delta.length === 0) return existing
  if (existing.length === 0) return delta

  // Cursor Composer can replay a complete text/output frame as the next delta.
  // If the current value already ends with that complete frame, appending it
  // produces the exact adjacent duplication shown in assistant output.
  if (existing.endsWith(delta)) {
    return existing
  }

  return existing + delta
}
```

```typescript
const MIN_DUPLICATE_TEXT_PART_LENGTH = 32;

export const collapseExactDuplicateAdjacentTextParts = (parts: Part[]): Part[] => {
    const collapsed: Part[] = [];
    let previousText: string | null = null;

    for (const part of parts) {
        if (part.type !== 'text') {
            collapsed.push(part);
            previousText = null;
            continue;
        }

        const text = extractTextContent(part).trim();
        if (
            text.length >= MIN_DUPLICATE_TEXT_PART_LENGTH &&
            previousText === text
        ) {
            continue;
        }

        collapsed.push(part);
        previousText = text.length > 0 ? text : null;
    }

    return collapsed;
};
```

## Risk Notes

- Do not dedupe reasoning parts. The reported bug is final assistant output, and reasoning can legitimately repeat intermediate thoughts.
- Do not globally dedupe all repeated words or sentences. Limit the guard to exact duplicate full-frame text/output chunks and exact duplicate adjacent assistant text parts.
- Keep the length threshold if the existing reducer test for `haha` fails. Cursor’s bad frames are sentence/tool-error sized; short repeated tokens must remain valid model output.
- Do not modify Cursor ACP provider configuration or OpenCode lifecycle for this fix. The output can originate upstream, but DevRyan should be robust to duplicate stream frames at the UI boundary.
